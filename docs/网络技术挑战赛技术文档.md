# ClawGuard Web 技术文档

## 一、项目背景

随着大模型智能体、自动化运维平台和安全分析工具逐渐进入真实业务环境，系统已经不再只是“接收输入、返回结果”的传统软件，而是会根据用户指令调用工具、读取文件、执行任务、访问网络甚至触发外部系统动作。能力边界扩大以后，安全边界也随之变得更加复杂。

OpenClaw 类工具具备较强的任务执行能力，适合用于安全实验、自动化分析和教学演示。但如果直接暴露任务入口，系统容易面临以下风险：

- 用户身份不可信，无法证明任务确由合法用户提交。
- 请求内容可被篡改，任务参数和输入文件缺少可信校验。
- 用户可能绕过界面提交危险命令或恶意参数。
- 任务执行环境权限过大，异常任务可能影响宿主机。
- 安全事件缺乏完整审计，事后难以追踪责任与复盘攻击路径。

因此，本项目围绕“可信身份、可信请求、安全执行、全链路审计”构建 ClawGuard Web。它不是直接修改 OpenClaw 内核，而是在 OpenClaw 外围增加一个 Web 安全代理层，将任务进入执行环境之前的认证、过滤、授权、沙箱和审计全部前置。

## 二、问题切入

### 2.1 核心问题

本项目聚焦的问题是：如何让一个具备任务执行能力的系统，在开放给多用户使用时，仍然满足身份可信、输入可控、权限可管、执行隔离和审计可追踪。

具体拆解为五类问题：

1. 身份可信问题  
   普通账号密码只能证明用户知道密码，不能证明关键操作确实由可信硬件或授权主体发起。

2. 输入安全问题  
   上传文件、任务参数和文本输入中可能包含私钥、token、路径穿越、内网地址或敏感配置。

3. 命令安全问题  
   如果允许用户直接提交 shell 命令，系统会暴露 `rm`、`curl`、`ssh`、`docker`、管道符、重定向等高危操作面。

4. 执行隔离问题  
   OpenClaw 执行任务时如果直接运行在宿主机，会带来文件系统、网络、进程和资源滥用风险。

5. 审计追踪问题  
   任务执行前后的认证、过滤、授权、容器状态和异常告警如果没有统一记录，系统无法形成安全闭环。

### 2.2 传统方案不足

传统 Web 项目常见做法是“登录后提交任务，后端直接执行脚本”。这种方案实现简单，但安全能力不足：

- 登录与关键任务提交之间缺少二次可信认证。
- 参数校验多停留在格式层面，无法识别安全语义风险。
- 权限控制往往只判断角色，缺少任务上下文和 Docker 策略联动。
- 执行环境和 Web 服务环境耦合，攻击面过大。
- 日志只记录业务结果，不记录安全判断过程。

因此，ClawGuard Web 的设计重点不是单点防护，而是建立一条从入口到执行再到审计的完整安全链路。

## 三、问题解决方案

### 3.1 总体思路

ClawGuard Web 采用“安全代理 + 策略控制 + 沙箱执行 + 审计闭环”的总体方案。

系统主线如下：

```text
用户 / 前端页面
    ↓
Web API 服务
    ↓
JWT 登录认证 / CFL UKey 签名认证
    ↓
输入过滤器 + 命令过滤器
    ↓
PBAC 权限策略判断
    ↓
Docker 沙箱执行 OpenClaw
    ↓
日志脱敏 / 审计追踪 / 安全告警
    ↓
结果返回前端
```

该方案将风险控制点分布在任务生命周期的关键位置：

- 任务进入前：身份认证、签名校验、Nonce 防重放。
- 任务解析时：输入过滤、命令过滤、危险参数拦截。
- 任务执行前：PBAC 策略判断、Docker 策略生成。
- 任务执行中：容器资源限制、网络隔离、只读文件系统。
- 任务执行后：结果收集、日志脱敏、审计记录、异常告警。

### 3.2 技术架构

系统分为五层：

1. 前端交互层  
   提供登录、任务提交、任务结果、审计日志、策略查看和 CFL 状态诊断页面。

2. Web API 服务层  
   使用 FastAPI 提供统一 API 入口，负责认证、调度、过滤、授权、任务管理和审计查询。

3. 安全认证与权限控制层  
   包含 JWT、CFL UKey、Nonce 防重放、输入过滤、命令过滤和 PBAC 策略。

4. OpenClaw 沙箱执行层  
   使用 Docker 镜像 `openclaw-sandbox:latest` 运行 OpenClaw stub 或未来真实 OpenClaw。

5. 日志审计与监控层  
   记录登录、验签、过滤、授权、任务、容器和告警事件，并对敏感字段脱敏。

## 四、核心实现方法

### 4.1 身份认证与 CFL 可信接入

系统支持两种认证模式：

- 普通账号密码登录。
- CFL UKey 签名登录。

普通登录用于基础访问控制，CFL 用于关键操作的可信身份增强。CFL 登录流程如下：

```text
用户请求 challenge
    ↓
后端生成 nonce + timestamp
    ↓
客户端使用 CFL UKey 签名
    ↓
后端校验 nonce、timestamp、签名、公钥
    ↓
生成 JWT Token
```

真实 CFL 接入采用两层兼容设计：

- `CFLRealService`：Python 后端真实 CFL 服务封装。
- `CflHelper.exe`：32 位本地 helper，用于桥接教学版 x86 `CFLClientLib.dll`。

由于教学版 `CFLClientLib.dll` 是 x86 DLL，而后端 Python 是 64 位，项目没有强行让 64 位 Python 直接加载 DLL，而是通过 32 位 `.NET Framework` helper 调用真实 DLL，再由后端通过子进程 JSON 协议调用 helper。

当前已实现：

- DLL 架构识别。
- helper 自动桥接。
- `CFL_Connect` 调用诊断。
- `CFL_GenRandom`、`CFL_ExportPublicKey`、`CFL_Digest`、`CFL_Sign`、`CFL_Verify` 封装。
- `/api/auth/cfl/status` 状态接口。
- `/api/auth/cfl/diagnostics` 深度诊断接口。

这使系统具备真实 CFL 接入能力，同时在无 UKey 或教学设备未初始化时仍可通过 mock 模式完整演示业务流程。

### 4.2 输入过滤器

输入过滤器用于扫描任务参数、文本输入和上传文件，识别高风险内容。

检测内容包括：

- 私钥片段：`-----BEGIN PRIVATE KEY-----`
- 云厂商 Access Key：`AKIA...`
- 敏感字段：`password`、`token`、`secret`、`api_key`
- 路径穿越：`../`
- 本地敏感路径：`/etc/passwd`、`~/.ssh`
- 内网与元数据地址：`127.0.0.1`、`localhost`、`169.254.169.254`、`10.0.0.0/8`

过滤结果分为三类：

- `ALLOW`：正常通过。
- `MASK`：允许执行，但日志脱敏。
- `DENY`：拒绝执行并记录高风险审计。

### 4.3 命令过滤器

系统不允许用户提交任意 shell 命令。用户只能选择预定义 action：

- `run_openclaw`
- `analyze_file`
- `export_result`

后端根据 action 映射到固定命令模板，并使用数组形式传递命令，不使用 `shell=True`。

命令过滤器拦截：

- `rm`
- `sudo`
- `chmod`
- `curl`
- `wget`
- `ssh`
- `scp`
- `docker`
- `bash`
- `sh`
- `powershell`
- `cmd.exe`
- `;`
- `&&`
- `||`
- `|`
- `>`
- `<`
- `` ` ``
- `$()`

这样可以把任务能力收敛到可审计、可控制、可授权的 action 范围内。

### 4.4 PBAC 权限策略

项目采用 PBAC，即 Policy-Based Access Control。

策略包含：

- Subject：用户、角色。
- Resource：任务、API、审计、策略。
- Action：执行、查看、导出、管理。
- Context：任务类型、风险等级、Docker 执行策略。

角色示例：

- `student`：可执行普通任务、查看结果，不可查看审计和管理策略。
- `admin`：可执行任务、查看审计、管理策略。
- `auditor`：只允许查看审计日志。

PBAC 不只决定“是否允许”，还会返回 Docker 执行策略，例如：

- 是否允许网络。
- 内存限制。
- CPU 限制。
- 进程数限制。
- 最大运行时间。

这使权限控制从 API 层延伸到了执行环境层。

### 4.5 Docker 沙箱执行

每个任务生成独立 `job_id`，并创建独立目录：

```text
jobs/
  job_xxx/
    input/
    output/
    logs/
```

容器执行策略包括：

- 非 privileged。
- 非 root 用户。
- 只读文件系统。
- 输入目录只读挂载。
- 输出目录读写挂载。
- `cap_drop=["ALL"]`
- `no-new-privileges`
- CPU 限制。
- 内存限制。
- 进程数限制。
- 网络默认禁用。
- 超时后自动终止。

这部分解决了 OpenClaw 执行能力过强带来的宿主机风险。

### 4.6 审计与告警

系统审计日志覆盖完整安全链路：

- `LOGIN_SUCCESS`
- `LOGIN_FAILED`
- `CFL_VERIFY_SUCCESS`
- `CFL_VERIFY_FAILED`
- `INPUT_FILTER_DENY`
- `COMMAND_FILTER_DENY`
- `POLICY_ALLOW`
- `POLICY_DENY`
- `TASK_CREATED`
- `TASK_SUCCESS`
- `TASK_FAILED`
- `TASK_TIMEOUT`
- `CONTAINER_ERROR`
- `ALERT_TRIGGERED`

审计记录包括：

- 用户。
- 角色。
- 事件类型。
- action。
- job_id。
- 风险等级。
- 执行结果。
- 脱敏后的 detail。

敏感字段写入日志前会统一脱敏，避免审计系统本身成为敏感信息泄漏点。

## 五、实现效果

当前项目已经完成并验证以下能力：

1. 后端 FastAPI 服务可运行。
2. 前端控制台可访问。
3. 管理员账号可登录。
4. 普通任务可提交。
5. Docker 沙箱镜像可构建并执行。
6. `run_openclaw` 任务可在沙箱中运行并返回结果。
7. 输入过滤器可拦截私钥、路径穿越和高风险地址。
8. 命令过滤器可拦截危险命令和 shell 符号。
9. PBAC 可限制普通用户访问审计和策略接口。
10. 审计日志可查询。
11. CFL x86 DLL 已通过 helper 桥接到 64 位后端。
12. CFL 状态和诊断接口可返回真实 DLL、helper、读卡器和错误码状态。

真实测试结果：

- `GET /api/auth/cfl/status` 显示 `mode=real`、`using_helper=true`、`dll_loadable=true`。
- Windows 可识别 `JNZG CCID KEY 0/1` 智能卡读卡器。
- 厂商 `FunTest.exe` 的导出公钥路径输出全零公钥。
- ClawGuard CFL 诊断接口进一步给出原始错误码：
  - `CFL_Connect = 167772197 (0x0A000025)`
  - `CFL_ExportPublicKey = 167772165 (0x0A000005)`

这说明项目已经打通到真实 CFL DLL，但当前 UKey 的 CFL 应用/容器环境仍需通过厂商工具进一步初始化。

## 六、项目特色与创新点

### 6.1 安全代理而非内核侵入

项目不修改 OpenClaw 内核，而是在外层增加安全代理。这样既保留了 OpenClaw 原有能力，又能快速增加认证、过滤、授权、沙箱和审计能力。

### 6.2 CFL 可信身份与 Web 安全结合

项目将 CFL UKey 引入 Web 任务系统，使任务提交不只依赖账号密码，还可以通过硬件签名证明身份和操作意图。

### 6.3 兼容 x86 教学 DLL 的真实接入方案

教学版 CFL DLL 是 x86，而 Web 后端通常运行在 x64 环境。项目通过 32 位 helper 桥接 x86 DLL，避免重构整个后端运行环境。这是工程上比较实用的兼容方案。

### 6.4 PBAC 与 Docker 策略联动

权限判断不仅决定接口是否允许访问，还会生成 Docker 执行策略。不同角色对应不同网络、CPU、内存、进程和超时限制，实现“权限控制到执行环境”的下沉。

### 6.5 入口过滤与过程审计闭环

输入过滤、命令过滤、权限判断和容器执行结果全部进入审计日志，形成“事前拦截、事中控制、事后追踪”的闭环。

### 6.6 可演示、可部署、可扩展

项目既支持 mock CFL 模式，也支持 real CFL 模式。在没有真实硬件或设备未初始化时，系统仍可演示完整安全流程；在硬件环境就绪后，可切换到真实签名验签。

## 七、项目价值

### 7.1 教学价值

项目适合用于网络安全、密码应用、系统安全和云原生安全教学，能够展示：

- 硬件可信认证如何接入 Web 系统。
- 用户输入如何进行安全过滤。
- 命令执行如何收敛到 action 模型。
- Docker 如何实现任务隔离。
- 审计日志如何支撑安全复盘。

### 7.2 工程价值

项目提供了一套轻量级安全代理架构，可用于改造已有自动化工具、智能体执行系统和教学实验平台。

它的优势是：

- 不侵入原有执行内核。
- 可独立部署。
- 可渐进增强安全能力。
- 可根据策略扩展角色和执行限制。

### 7.3 安全价值

项目通过认证、过滤、授权、隔离和审计多层控制，降低了任务执行系统的风险：

- 降低非法用户提交任务的风险。
- 降低危险命令执行风险。
- 降低敏感信息泄漏风险。
- 降低任务逃逸宿主机风险。
- 提高攻击溯源和安全复盘能力。

## 八、未来展望

后续可以从以下方向继续增强：

1. CFL 设备初始化适配  
   对接厂商 UKey 初始化工具，自动检测应用、容器、公钥和 PIN 状态。

2. CFL 公钥登记流程  
   增加用户公钥绑定、管理员审核和公钥轮换机制。

3. 策略可视化管理  
   将 YAML PBAC 策略升级为 Web 可视化编辑，并加入策略版本控制。

4. 文件上传安全增强  
   增加文件类型识别、压缩包检测、恶意样本隔离和内容哈希库。

5. 沙箱执行增强  
   引入 seccomp、AppArmor、只读 rootfs 白名单和更细粒度网络策略。

6. 报告输出能力  
   生成安全任务报告、审计报告和异常告警报告，适配比赛展示和教学归档。

7. 多租户能力  
   支持班级、队伍、实验环境隔离，适合竞赛平台和教学平台部署。

## 九、总结提炼

ClawGuard Web 面向 OpenClaw 任务执行场景，解决的是“强执行能力如何安全开放”的问题。

项目以身份可信为起点，以输入和命令过滤为入口控制，以 PBAC 为权限核心，以 Docker 为执行边界，以审计日志为闭环支撑，最终形成：

```text
可信认证 -> 安全过滤 -> 策略授权 -> 沙箱隔离 -> 审计追踪
```

相比传统任务执行平台，ClawGuard Web 的核心价值在于将密码硬件、Web 安全、权限策略和容器隔离融合到同一条任务链路中。它不仅能演示安全机制，也具备真实工程落地的结构基础，适合网络技术挑战赛中展示系统设计能力、工程实现能力和安全创新能力。
